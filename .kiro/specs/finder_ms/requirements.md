# 요구사항 문서

## 소개

본 문서는 분실물 찾아주기 플랫폼(Lost & Found AI Matching Platform)의 요구사항을 정의한다. 이 플랫폼은 비전 및 텍스트 기반 AI를 활용하여, 발견자가 습득한 분실물을 등록하면 AI가 자동으로 이미지 캡션을 생성하고, 찾는 사람의 텍스트 설명과 유사도를 비교하여 매칭해주는 시스템이다.

## 용어 정의

- **플랫폼**: 분실물 찾아주기 AI 매칭 플랫폼 전체 시스템
- **발견자**: 분실물을 습득하여 플랫폼에 등록하는 사용자
- **찾는자**: 분실물을 잃어버려 플랫폼에서 검색하는 사용자
- **분실물_게시물**: 발견자가 등록한 분실물 정보(사진, 카테고리, 장소, 시간, AI 캡션 포함)
- **캡션_생성기**: 분실물 이미지를 입력받아 텍스트 캡션을 자동 생성하는 AI 비전 모델 모듈
- **매칭_엔진**: 찾는자의 텍스트 설명과 분실물_게시물의 AI 캡션 간 유사도를 계산하여 매칭 결과를 반환하는 모듈
- **유사도_점수**: 매칭_엔진이 산출하는 0~1 범위의 텍스트 유사도 수치
- **카테고리**: 분실물 분류 체계 (지갑, 카드, 신분증, 전자기기, 기타 등)
- **채팅_서비스**: 발견자와 찾는자 간 실시간 메시지를 중계하는 WebSocket 기반 모듈
- **본인인증_서비스**: 전화번호, 주민등록번호, 학교 이메일 등을 통해 사용자 신원을 검증하는 모듈
- **이름_매칭_검증**: 신분증류 분실물에 대해 찾는자의 등록 이름과 분실물 소유자 이름의 일치 여부를 확인하는 절차

## 요구사항

### 요구사항 1: 분실물 등록

**사용자 스토리:** 발견자로서, 습득한 분실물을 사진과 함께 등록하고 싶다. 그래야 찾는 사람이 해당 분실물을 발견할 수 있다.

#### 인수 조건

1. WHEN 발견자가 분실물 사진을 업로드하면, THE 플랫폼 SHALL 해당 사진을 저장하고 분실물_게시물을 생성한다
2. WHEN 발견자가 분실물을 등록할 때, THE 플랫폼 SHALL 카테고리(지갑, 카드, 신분증, 전자기기, 기타) 중 하나를 선택하도록 요구한다
3. WHEN 발견자가 분실물을 등록할 때, THE 플랫폼 SHALL 발견 장소와 발견 시간을 입력받는다
4. WHEN 분실물 사진이 업로드되면, THE 캡션_생성기 SHALL 해당 이미지에 대한 텍스트 캡션을 자동으로 생성하여 분실물_게시물에 저장한다
5. IF 사진 업로드가 실패하면, THEN THE 플랫폼 SHALL 발견자에게 업로드 실패 사유를 포함한 오류 메시지를 반환한다
6. IF 캡션_생성기가 캡션 생성에 실패하면, THEN THE 플랫폼 SHALL 발견자에게 수동 캡션 입력을 요청한다

### 요구사항 2: AI 이미지 캡션 생성

**사용자 스토리:** 발견자로서, 분실물 사진을 올리면 AI가 자동으로 설명을 생성해주길 원한다. 그래야 찾는 사람이 텍스트 검색으로 매칭할 수 있다.

#### 인수 조건

1. WHEN 유효한 이미지 파일이 입력되면, THE 캡션_생성기 SHALL 해당 이미지의 시각적 특징을 설명하는 텍스트 캡션을 생성한다
2. THE 캡션_생성기 SHALL 생성된 캡션을 한국어 텍스트로 반환한다
3. WHEN 캡션이 생성되면, THE 플랫폼 SHALL 발견자에게 생성된 캡션을 표시하고 수정할 수 있는 기회를 제공한다
4. IF 지원하지 않는 이미지 형식이 입력되면, THEN THE 캡션_생성기 SHALL 지원 형식 목록을 포함한 오류 메시지를 반환한다

### 요구사항 3: 분실물 검색 및 AI 매칭

**사용자 스토리:** 찾는자로서, 잃어버린 물건의 특징을 텍스트로 설명하면 AI가 유사한 분실물을 찾아주길 원한다. 그래야 빠르게 분실물을 찾을 수 있다.

#### 인수 조건

1. WHEN 찾는자가 텍스트 설명을 입력하면, THE 매칭_엔진 SHALL 등록된 분실물_게시물의 캡션과 유사도_점수를 계산한다
2. WHEN 매칭 결과가 산출되면, THE 매칭_엔진 SHALL 유사도_점수가 높은 순서로 분실물_게시물 목록을 반환한다
3. THE 매칭_엔진 SHALL 각 매칭 결과에 0에서 1 사이의 유사도_점수를 포함한다
4. WHEN 찾는자가 카테고리를 선택하면, THE 플랫폼 SHALL 해당 카테고리에 속하는 분실물_게시물만 필터링하여 표시한다
5. IF 매칭 결과가 없으면, THEN THE 플랫폼 SHALL 찾는자에게 일치하는 분실물이 없음을 안내하고 알림 등록을 제안한다

### 요구사항 4: 카테고리별 브라우징

**사용자 스토리:** 찾는자로서, 카테고리별로 분실물 목록을 탐색하고 싶다. 그래야 AI 매칭 외에도 직접 분실물을 찾아볼 수 있다.

#### 인수 조건

1. THE 플랫폼 SHALL 등록된 분실물_게시물을 카테고리별로 분류하여 목록을 제공한다
2. WHEN 찾는자가 특정 카테고리를 선택하면, THE 플랫폼 SHALL 해당 카테고리의 분실물_게시물을 최신 등록순으로 표시한다
3. WHEN 찾는자가 분실물_게시물을 선택하면, THE 플랫폼 SHALL 사진, 캡션, 카테고리, 발견 장소, 발견 시간을 포함한 상세 정보를 표시한다

### 요구사항 5: 채팅 기능

**사용자 스토리:** 찾는자로서, 분실물 발견자와 실시간으로 대화하고 싶다. 그래야 분실물 확인 및 수령 일정을 조율할 수 있다.

#### 인수 조건

1. WHEN 찾는자가 특정 분실물_게시물에서 채팅을 요청하면, THE 채팅_서비스 SHALL 발견자와 찾는자 간 1:1 채팅방을 생성한다
2. WHILE 채팅방이 활성 상태인 동안, THE 채팅_서비스 SHALL WebSocket을 통해 실시간으로 메시지를 전달한다
3. THE 채팅_서비스 SHALL 전송된 모든 메시지를 발신자 정보, 타임스탬프와 함께 저장한다
4. IF WebSocket 연결이 끊어지면, THEN THE 채팅_서비스 SHALL 자동으로 재연결을 시도하고, 연결 복구 시 미수신 메시지를 전달한다

### 요구사항 6: 만남 예약

**사용자 스토리:** 발견자와 찾는자로서, 분실물 교환을 위한 만남 일정을 예약하고 싶다. 그래야 원활하게 분실물을 수령할 수 있다.

#### 인수 조건

1. WHEN 채팅 참여자 중 한 명이 만남 예약을 요청하면, THE 플랫폼 SHALL 만남 장소와 시간을 입력받는 예약 양식을 제공한다
2. WHEN 만남 예약이 생성되면, THE 플랫폼 SHALL 상대방에게 예약 확인 요청 알림을 전송한다
3. WHEN 상대방이 예약을 수락하면, THE 플랫폼 SHALL 양측 모두에게 확정된 만남 일정을 표시한다
4. IF 상대방이 예약을 거절하면, THEN THE 플랫폼 SHALL 요청자에게 거절 사실을 알리고 재예약을 제안한다

### 요구사항 7: 찾기완료 상태 관리

**사용자 스토리:** 발견자와 찾는자로서, 분실물 교환이 완료되면 상태를 표시하고 싶다. 그래야 다른 사용자가 이미 찾은 물건을 검색하지 않는다.

#### 인수 조건

1. WHEN 발견자 또는 찾는자가 "찾기완료"를 요청하면, THE 플랫폼 SHALL 상대방에게 완료 확인 요청을 전송한다
2. WHEN 양측 모두 완료를 확인하면, THE 플랫폼 SHALL 해당 분실물_게시물의 상태를 "찾기완료"로 변경한다
3. WHILE 분실물_게시물의 상태가 "찾기완료"인 동안, THE 플랫폼 SHALL 해당 게시물을 매칭 검색 결과에서 제외한다
4. THE 플랫폼 SHALL "찾기완료" 상태의 게시물을 카테고리 브라우징에서 별도로 구분하여 표시한다

### 요구사항 8: 회원가입

**사용자 스토리:** 사용자로서, 플랫폼에 회원가입하고 싶다. 그래야 분실물 등록 및 검색 서비스를 이용할 수 있다.

#### 인수 조건

1. WHEN 사용자가 회원가입을 요청하면, THE 플랫폼 SHALL 이름, 전화번호, 비밀번호를 입력받는다
2. THE 플랫폼 SHALL 비밀번호를 해시 처리하여 저장한다
3. WHEN 회원가입이 완료되면, THE 플랫폼 SHALL 사용자에게 인증 토큰을 발급한다
4. IF 이미 등록된 전화번호로 가입을 시도하면, THEN THE 플랫폼 SHALL 중복 가입 불가 안내 메시지를 반환한다

### 요구사항 9: 본인인증

**사용자 스토리:** 사용자로서, 본인인증을 완료하고 싶다. 그래야 플랫폼의 신뢰성이 보장되고 안전하게 분실물을 교환할 수 있다.

#### 인수 조건

1. WHEN 사용자가 전화번호 인증을 요청하면, THE 본인인증_서비스 SHALL SMS로 인증 코드를 전송한다
2. WHEN 사용자가 올바른 인증 코드를 입력하면, THE 본인인증_서비스 SHALL 전화번호 인증 완료 상태를 기록한다
3. IF 인증 코드가 일치하지 않으면, THEN THE 본인인증_서비스 SHALL 인증 실패 메시지를 반환하고 재시도를 허용한다
4. IF 인증 코드 입력 시도가 5회를 초과하면, THEN THE 본인인증_서비스 SHALL 해당 인증 요청을 차단하고 30분 후 재시도를 안내한다
5. WHEN 사용자가 학교 인증을 요청하면, THE 본인인증_서비스 SHALL 학교 이메일로 인증 링크를 전송한다
6. WHEN 사용자가 인증 링크를 클릭하면, THE 본인인증_서비스 SHALL 학교 인증 완료 상태를 기록한다

### 요구사항 10: 신분증류 분실물 이름 매칭 검증

**사용자 스토리:** 찾는자로서, 신분증류 분실물(지갑, 카드, 신분증)을 요청할 때 본인 확인을 거치고 싶다. 그래야 타인의 신분증이 부정하게 전달되는 것을 방지할 수 있다.

#### 인수 조건

1. WHEN 찾는자가 카테고리가 "신분증", "카드", "지갑" 중 하나인 분실물_게시물에 대해 채팅을 요청하면, THE 플랫폼 SHALL 이름_매칭_검증 절차를 시작한다
2. WHEN 이름_매칭_검증이 시작되면, THE 플랫폼 SHALL 찾는자의 회원가입 시 등록한 이름과 분실물 소유자 이름의 일치 여부를 확인한다
3. IF 이름이 일치하지 않으면, THEN THE 플랫폼 SHALL 채팅 요청을 거부하고 본인 소유 분실물만 요청 가능하다는 안내 메시지를 반환한다
4. IF 분실물_게시물에 소유자 이름 정보가 없으면, THEN THE 플랫폼 SHALL 이름_매칭_검증을 건너뛰고 채팅을 허용한다

### 요구사항 11: API 응답 형식

**사용자 스토리:** 개발자로서, 일관된 JSON 형식의 API 응답을 받고 싶다. 그래야 클라이언트 개발 시 예측 가능한 데이터 구조를 활용할 수 있다.

#### 인수 조건

1. THE 플랫폼 SHALL 모든 API 응답을 JSON 형식으로 반환한다
2. THE 플랫폼 SHALL 모든 API 응답에 성공 여부(success), 데이터(data), 메시지(message) 필드를 포함한다
3. THE 플랫폼 SHALL API 응답의 메시지 필드를 한국어로 제공한다
4. IF API 요청 처리 중 오류가 발생하면, THEN THE 플랫폼 SHALL HTTP 상태 코드와 함께 오류 유형(error_code)과 한국어 오류 메시지를 반환한다
